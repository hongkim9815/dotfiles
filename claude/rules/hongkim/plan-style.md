## 불변 원칙 (system-level)

- 언어: 한국어 (코드·경로·명령어 제외)
- 독자: AI — Human 독자용 원칙 적용 금지
- 자기완결성: 각 섹션 단독 발췌 시 의미 통해야 함
- 산문 금지: 불릿·표 우선, 중요 정보 상단 배치

---

## 발동 조건 (Scheduling)

허용:
- 사용자가 "plan 짜줘", "계획 세워줘", "plan 써줘" 등 **명시적으로 plan 작성을 요청**한 경우

금지:
- 암시적 요청("이거 어떻게 해?", "뭐가 문제야?") → plan 작성 금지, 답변만 제공
- 사용자 승인 없이 탐색 단계에서 바로 plan 작성 금지

---

## 실행 흐름 (Structural)

**1단계 — 탐색·수집**:
- 코드베이스 탐색, 질문, 맥락 수집
- 정보 불충분 → `AskUserQuestion` 으로 질의 후 대기
- plan 파일 작성 금지

**2단계 — 작성**:
- 필수 섹션 구조에 따라 작성
- 완료 기준: 배경·작업·핵심 파일·검증 4개 섹션 모두 작성 완료 + 각 섹션 자기완결성 충족
- `./.claude/plans/` 에 저장 (`Write` 도구 사용)
- 저장 완료 후 대기 — plan 모드 종료는 Claude Code 자체 기능에 위임

---

## 저장 위치

plan 파일은 반드시 **현재 작업 중인 repository의 `.claude/plans/`** 에 저장.

- 저장 경로 패턴: `./.claude/plans/**`
- 파일명 형식: `./.claude/plans/{ISSUE-ID}-{kebab-case-title}.md`
- ISSUE-ID 없는 경우: `./.claude/plans/{kebab-case-title}.md`
- 예시: `./.claude/plans/PROJ-123-test-db-migration.md`
- `.`은 현재 작업 디렉토리 기준 — 절대경로·전역경로 사용 금지
- `~/.claude/plans/` 또는 다른 전역 경로에 저장 금지

## 필수 섹션 구조

```
# Plan: [작업 제목]

## 배경
## 작업
## 핵심 파일
## 검증
```

각 섹션은 단독 발췌해도 의미가 통해야 함.

---

## 배경 섹션

- 이 변경이 왜 필요한지 — 문제·요구사항 기술
- 무엇을 해결하는지 — 의도된 결과 명시
- 배경 정보 — 관련 파일, 기존 패턴, 제약 조건

## 작업 섹션

- 실행 단계를 번호 순서로 나열
- 각 단계: 동사 시작 (`생성`, `수정`, `삭제`, `실행`)
- 조건 선행 명시: `X 상태일 때 → Y 실행`
- 병렬 실행 가능 단계는 명시: `(병렬 가능)`

## 핵심 파일 섹션

수정·생성·삭제 파일을 표로 정리:

| 파일 경로 | 작업 | 재사용할 기존 함수/패턴 |
|---|---|---|
| `path/to/file.ts` | 수정 | `existingUtil()` at line 42 |

## 검증 섹션

- 변경 후 검증 방법 (실행 명령, MCP 도구, 테스트)
- 성공 기준 명시 (기대 출력, 상태 코드, 커버리지)
- 실패 시 처리 명시: `검증 실패 시 → {롤백 방법 또는 재시도 조건}`

---

## 금지 사항

- 대안 여러 개 나열 금지 — 권장 접근법 하나만 작성
- 불필요한 배경 설명 금지 — 실행에 필요한 정보만 포함
- 추측성 내용 금지 — 확인된 정보만 기술
- 높임말 어미 금지 (`~합니다`, `~주세요`)
