# Testing Requirements

## 테스트 전략

통합 테스트 우선. 환경 제약으로 불가능한 경우에만 unit test로 대체.

| 상황 | 적용 | Mock 정책 |
|---|---|---|
| DB·API·외부 서비스 연동 가능 | Integration Test | 외부 의존 mock 금지, 실제 사용 |
| 환경 구성 불가 (sandbox 없음, CI 미지원) | Unit Test | 외부 의존만 mock, 내부 로직 mock 금지 |
| E2E (UI 플로우) | 핵심 경로만 | 백엔드 실제 사용, 외부 결제·이메일만 mock |

**핵심 경로 정의**: 사용자 가치 90%+ 차지하는 플로우 (로그인·결제·핵심 CRUD). 부가 기능 제외.

## 테스트 구성 원칙

수치 커버리지 목표 없음. 요건 기반으로 테스트 케이스를 도출할 것.

- **요건 → 테스트 케이스** 매핑: 각 요건/시나리오에 대응하는 테스트 존재해야 함
  - **언어별 산출 형식**:

    | 언어 | 형식 |
    |---|---|
    | TypeScript/JavaScript | 파일 상단 JSDoc 블록 (`@requirement REQ-001`) |
    | Java/Kotlin | 클래스 상단 JavaDoc 블록 |
    | Python | 모듈 docstring 첫 줄 또는 test class docstring |
    | Go | `t.Run("REQ-001/정상: ...", func(t *testing.T) { ... })` |
    | Ruby/Rust | 파일 상단 주석 블록 또는 `describe "REQ-001"` (RSpec) |

    예시 (TS):
    ```
    /**
     * 요건: REQ-001 사용자 로그인 성공
     * 케이스: 정상 / 잘못된 비밀번호 / 계정 잠금 / 토큰 만료
     */
    ```
- **정상 케이스** 외에 아래를 포함:
  - 경계값 (boundary values)
  - 예외/에러 케이스 (invalid input, 외부 서비스 실패 등)
  - 비즈니스 규칙 위반 케이스
- 테스트 케이스 누락 시 요건 누락으로 간주

## 테스트 실패 트러블슈팅

**Loop (최대 3회)**:
1. 테스트 격리 확인 (단독 실행 시 통과 여부)
2. mock 정확성 검증 (실제 동작과 일치 여부)
3. 원인 분류 후 분기:
   - 구현 버그 → 구현 수정 → 재실행
   - 테스트 오류 (요건 변경 미반영·잘못된 전제) → 테스트 수정 → 재실행
   - 환경 문제 (DB·외부 서비스) → 환경 복구 → 재실행

**3회 임계값 근거**: 동일 가설 4회차부터는 새로운 정보 없이 반복할 확률이 높음. 사용자 개입으로 가설 확장 필요.

**에스컬레이션**: 3회 반복 후 미해결 시 → 사용자에게 보고:
- 시도한 가설 목록
- 각 시도 결과
- 추정 원인 후보 (가능성 순)
- 추가로 필요한 정보·접근 권한
