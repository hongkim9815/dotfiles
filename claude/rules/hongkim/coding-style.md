# Coding Style

## 객체지향 원칙

**적용 시점**: 신규 클래스·모듈 설계 시. 기존 코드는 리팩터링 요청 있을 때만.

| 원칙 | 정의 | Anti-pattern |
|---|---|---|
| SRP | 클래스·함수는 하나의 책임만 | God object, 200줄+ 함수 |
| OCP | 확장 개방, 수정 폐쇄 | switch/if 분기로 타입 처리 |
| LSP | 하위 타입이 상위 타입 완전 대체 가능 | 하위 타입에서 NotImplementedError |
| ISP | 미사용 메서드 강제 인터페이스 금지 | Fat interface |
| DIP | 추상 의존, 구체 의존 금지 | `new ConcreteClass()` 직접 호출 |
| DI | 외부 주입, 내부 생성 금지 | 생성자 내 `new` |

---

## 불변성

**기본**: 객체 직접 수정 금지 → 새 객체 반환.

**예외 (mutation 허용)**:
- 성능 critical hot path (벤치마크로 입증 시)
- 빌더 패턴 내부 (외부 노출 전)
- 명시적 가변 컬렉션 (Mutable* 타입)

---

## 단순성 원칙 (Simplicity First)

**충돌 시 우선순위**: 단순성 > 객체지향 원칙 (특히 OCP). 추상화는 3회 반복 후만.

- 조기 추상화 금지 — 3개 이상 반복될 때 추상화 고려
- 발생하지 않는 시나리오에 대한 에러 처리 금지
- 미래 요구사항을 위한 확장 포인트 선제 추가 금지

---

## 파일 구성

- 200~400줄 권장, 최대 800줄
- 기능/도메인별로 구성

---

## 작업 완료 체크리스트

**적용 시점**: 코드 작성·수정 완료 후, 완료 보고 직전.

**자동 검증 (Bash·grep으로 확인 가능)**:

| 항목 | 기준 | 검증 명령 |
|---|---|---|
| 함수 길이 | 50줄 이하 | `awk '/^(function\|def\|func)/,/^}/{c++} END{print c}'` |
| 파일 길이 | 800줄 이하 | `wc -l <file>` |
| 중첩 깊이 | 4레벨 이하 | `grep -E "^( {16,}\|\t{4,})" <file> \| wc -l` (0이어야 함) |

**수동 검토 (LLM 판단)**:
- [ ] 하드코딩 없음 (매직 넘버·시크릿·경로)
- [ ] 불변 패턴 사용

미충족 시 → 수정 후 재검증 → 통과 후 완료 보고. 미충족 완료 보고 금지.
