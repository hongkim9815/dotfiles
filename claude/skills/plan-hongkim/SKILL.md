# Plan

## 발동 조건 (MANDATORY)

허용:
- 사용자가 "plan 짜줘", "계획 세워줘", "plan 써줘" 등 **명시적으로 plan 작성을 요청**한 경우

금지:
- 암시적 요청("이거 어떻게 해?", "뭐가 문제야?") → plan 작성 금지, 답변만 제공
- 사용자 승인 없이 탐색 단계에서 바로 plan 작성 금지

---

## Workflow

**1단계 — 탐색·수집**:
- 코드베이스 탐색, 질문, 맥락 수집
- 정보 불충분 → `AskUserQuestion` 으로 질의 후 대기
- plan 작성 금지

**1.5단계 — 정보 충분성 게이트 (MANDATORY)**:
아래 모두 충족 시에만 2단계 진입:
- [ ] 변경 대상 파일 경로 식별됨 (추측 아님)
- [ ] 재사용할 기존 함수/패턴 1개 이상 확인됨
- [ ] 검증 방법(테스트/명령) 1개 이상 식별됨
미충족 시 → 1단계로 복귀.

**2단계 — 작성**:
- 아래 필수 섹션 구조에 따라 작성
- 완료 기준: 배경·작업·핵심 파일·검증 4개 섹션 모두 작성 + 각 섹션 자기완결성 충족
- 작성 완료 후 사용자에게 파일 저장 여부 확인
  - 저장 요청 시: `./.claude/plans/{kebab-case-title}.md` 에 저장 (`Write` 도구 사용)
  - 저장 불필요 시: 대화 내 출력으로 종료

---

## 작성 원칙

- 언어: 한국어 (코드·경로·명령어 제외)
- 독자: AI — Human 독자용 원칙 적용 금지
- 자기완결성: 각 섹션 단독 발췌 시 의미 통해야 함
- 산문 금지: 불릿·표 우선, 중요 정보 상단 배치

## 필수 섹션 구조

```
# Plan: [작업 제목]

## 배경
## 작업
## 핵심 파일
## 검증
```

### Task 유형별 섹션 가중치

| 유형 | 배경 | 작업 | 핵심 파일 | 검증 |
|---|---|---|---|---|
| 신규 기능 | 상세 | 상세 | 상세 | 상세 |
| 버그 수정 | 재현 경로 필수 | 간결 | 영향 파일 | 회귀 테스트 필수 |
| 리팩토링 | 변경 이유 명확 | 단계별 안전성 | 영향 범위 | 동등성 검증 |
| 설정 변경 | 변경 이유 | 간결 | 변경 파일 | 적용 확인 |

### 배경 섹션

- 이 변경이 왜 필요한지 — 문제·요구사항 기술
- 무엇을 해결하는지 — 의도된 결과 명시
- 관련 파일, 기존 패턴, 제약 조건

### 작업 섹션

각 단계는 아래 signature로 작성:

```
N. [동사] {대상}
   - 입력: {필요한 정보·파일}
   - 산출: {생성·수정 결과물}
   - 의존: {선행 단계 번호 또는 "없음"}
   - 병렬: {가능한 다른 단계 번호 또는 "없음"}
```

조건 선행 명시: `X 상태일 때 → Y 실행`

### 핵심 파일 섹션

| 파일 경로 | 작업 | 재사용할 기존 함수/패턴 |
|---|---|---|
| `path/to/file.ts` | 수정 | `existingUtil()` at line 42 |

### 검증 섹션

- 변경 후 검증 방법 (실행 명령, MCP 도구, 테스트)
- 성공 기준 명시 (기대 출력, 상태 코드, 커버리지)
- 실패 시 처리 명시: `검증 실패 시 → {롤백 방법 또는 재시도 조건}`

## 금지 사항

- 대안 여러 개 나열 금지 — 권장 접근법 하나만 작성
- 불필요한 배경 설명 금지 — 실행에 필요한 정보만 포함
- 추측성 내용 금지 — 확인된 정보만 기술
  - "확인된" 기준: Read/Grep/Bash로 직접 검증된 것
  - 추측 표기 의무: 검증 불가 항목은 `(미확인 — 검증 필요)` 명시
  - **후속 검증 강제**: `(미확인)` 표기 항목은 **검증 섹션에 해당 항목 검증 방법 별도 명시 의무**.
    - 미기재 시 plan 완료 게이트 미통과 처리.
    - 형식: `검증 섹션 하단에 "미확인 항목 검증" 서브섹션 추가 → 각 항목별 검증 명령·방법 기재`
- 높임말 어미 금지 (`~합니다`, `~주세요`)
