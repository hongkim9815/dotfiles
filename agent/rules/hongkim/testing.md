# Testing Requirements

## 테스트 전략

통합 테스트 우선. 환경 제약으로 불가능한 경우에만 unit test로 대체.

| 상황 | 적용 |
|---|---|
| DB, API, 외부 서비스 연동 가능 | Integration Test |
| 테스트 환경 구성 어려움 | Unit Test |
| E2E (UI 플로우) | 핵심 경로만, 선택적 |

## 테스트 구성 원칙

수치 커버리지 목표 없음. 요건 기반으로 테스트 케이스를 도출할 것.

- **요건 → 테스트 케이스** 매핑: 각 요건/시나리오에 대응하는 테스트 존재해야 함
- **정상 케이스** 외에 아래를 포함:
  - 경계값 (boundary values)
  - 예외/에러 케이스 (invalid input, 외부 서비스 실패 등)
  - 비즈니스 규칙 위반 케이스
- 테스트 케이스 누락 시 요건 누락으로 간주

## 테스트 실패 트러블슈팅

1. 테스트 격리 확인
2. mock 정확성 검증
3. 구현 수정 (테스트가 명백히 잘못된 경우가 아니라면 테스트 수정 금지)
